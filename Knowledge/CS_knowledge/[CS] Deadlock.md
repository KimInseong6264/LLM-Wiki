# 1. 정의 및 발생 원인

---

### 1) 3줄 요약
- **정의**: 프로세스들이 서로 자원을 기다리며 무한 대기하는 상태.
- **원인**: 상호배제, 점유대기, 비선점, 순환대기 4가지가 모두 충족될 때 발생.
- **대응**: 타조 알고리즘(무시)이 현대 OS의 일반적 선택.

### 2) 발생 4대 조건 (Coffman Conditions)
1. **상호 배제 (Mutual Exclusion)**: 자원은 한 번에 한 프로세스만 사용 가능.
2. **점유와 대기 (Hold and Wait)**: 자원 보유 중 다른 자원 추가 대기.
3. **비선점 (No Preemption)**: 타인의 자원 강제 탈취 불가.
4. **순환 대기 (Circular Wait)**: 대기 관계가 서로 물고 물리는 원형을 이룸.

### 3) 발생 구조 (순환 대기)
```text
    [Process A] ----> [Resource 1]
        ^                  |
        |                  |
        |                  v
    [Resource 2] <---- [Process B]
```

# 2. 대응 전략

---

### 1) 전략 요약
| 전략명 | 핵심 방식 |
| :--- | :--- |
| **예방** | 4대 조건 중 하나를 강제로 제거 |
| **회피** | 안전 상태인지 검사 후 할당 |
| **탐지/회복** | 주기적 감지 후 강제 종료 또는 선점 |
| **타조 알고리즘** | 발생 자체를 무시하고 방치 |

### 2) 상세 설명
- **타조 알고리즘 (Ostrich Algorithm)**: 데드락 예방/감지에 드는 오버헤드가 매우 클 때 사용하는 전략입니다. 타조가 위험을 피하기 위해 모래 속에 머리를 박는 모습에서 유래했습니다. 실제 대부분의 시스템에서 데드락은 드물게 발생하므로, 문제를 해결하기 위해 시스템 성능을 저하시키기보다 **그냥 무시하는 것이 비용 효율적**이라는 논리입니다.
- **은행원 알고리즘 (Banker's Algorithm)**: 고객(프로세스)에게 대출(자원 할당)할 때, 은행(OS)이 파산(데드락)하지 않도록 **시스템이 항상 안전 상태(Safe State)를 유지하는지 미리 계산**하여 할당하는 방식입니다. 하지만 자원의 최대 요구량을 미리 파악해야 하는 제약이 있습니다.
- **예방**: 4대 조건(상호배제, 점유대기, 비선점, 순환대기) 중 하나를 부정합니다. 예: 모든 자원을 한 번에 할당하거나, 자원에 우선순위를 매겨 순환을 방지합니다.
- **탐지 및 회복**: 시스템을 주기적으로 검사하여 데드락이 확인되면, 관련 프로세스를 하나씩 강제 종료하거나 자원을 강제로 뺏어(Preemption) 해결합니다.

# 3. 실전 예방 및 참고

---

### 1) 실전 예방 팁
- **Lock Ordering**: 여러 개의 Lock을 획득해야 할 때, **항상 동일한 순서**(예: 자원 1번 획득 후 2번 획득)로 락을 획득하도록 코딩하여 순환 대기를 원천 방지하는 가장 실용적인 기법입니다.
- **식사하는 철학자 문제**: 동기화 문제의 고전적 예시로, 철학자들이 포크를 한쪽씩만 집고 서로의 포크를 기다리는 상황을 통해 데드락 발생 메커니즘을 설명합니다.
